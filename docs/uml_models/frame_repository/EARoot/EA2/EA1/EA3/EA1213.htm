This Function shall be capable of providing the following facilities:<br/><br/>(1) The ability to receive and process a request from an entity that wants to become new NAP User.<br/>(2) The ability to check the new NAP User in (1) above to include electronic identification, verification that the requesting entity exists based on information included in the request and agree with terms of the NAP User (different for NAP Content Provider and NAP Content Consumer)<br/>(3) The ability to load information about the entity submitting the request in (1) above into the NAP Users data store, if the identification, verification, and acceptance in (2) above is successful.<br/>(4) The ability to provide authorisation credentials to the requesting entity.<br/>(5) The ability to send a rejection notice (including the reason for rejection) to the entity that submitted the request in (1) above if the identification, verification, and acceptance in (2) and (3) above was not successful.<br/>(6) The ability to check the authorisation of an entity which sends its credentials to the NAP functions and to response to those functions if the authorisation check has been successful.<br/>(7) The ability to process the information provided in an unsuccessful request for the update of NAP User data so that a suitable response can be sent to the entity that made the request and future unsuccessful update requests monitored.<br/>(8) The ability to register multiple user accounts within a NAP User account, i.e. users' organization.<br/>(9) The ability to register multiple machine accounts within a NAP User account, i.e. users' organization, for consuming or providing content from / to the NAP. The machine accounts stand for Data Requesting Systems and Data Supplying System.<br/>(10) The ability to organize user and machine account registered under one NAP User into organization, where user accounts have different rights to access the NAP Users data store, i.e. view, modify, add, and delete records.<br/>(11) The ability to remove electronic identity of the NAP User from the NAP Users data store based on reception of a request from a NAP User.<br/>(12) The ability to register the as one or both NAP Content Provider and NAP Content Consumer with different rights / responsibilities to perform tasks assigned to provider or consumer in the NAP via Functions Provide Interface for Content Providers and Provide Access for NAP Metadata Retrieval.<br/>(13) Ability to provide registered NAP Content Provider data upon a successful authorization <br/>NOTE: one physical user can be part of more NAP Users but under the different electronic identity.<br/><br/>=========<br/>CHANGELOG <br/>- Changed NAP data provider to NAP User,  <br/>- Added possibility to register NAP Content Consumers,  <br/>- Added possibility to change and delete NAP User records <br/>- Added possibility to organise registrations to include multiple user and machine accounts <br/>- Added functionality to provide CP data that are stored within the system. <br/> <br/>TODO: <br/>- Check if the overall functionality holds up. Especially if the functionality (4) and (6) should not be as a separate function. <br/>- Change name to Manage NAP User Datastore and add functionality to log in events connected with NAP User Activity to the NAP User data store <br/>- Check if NOTE is OK with everybody. <br/>- IMPORTANT: the function shall also return data objects based on the provided NAP User credentials for loading data to CP system or shall manage the registration data within<br/><br/>

(a) continuously monitor for the receipt of the first three logical input data flows;<br/>(b) if the first logical input data flow is received in (a), process the request and separate the data from the information about the data provider;<br/>(c) carry out any necessary checks to establish the existence, legitimacy and any other checks for the data provider to be able to submit data to either of the NAP data stores;<br/>(d) load the data found in (b) into the second output logical data flow and send it;<br/>(e) load the information about the data provider found in (b) into the first logical output data flow and send it;<br/>(f) if the second logical input data flow is received in (a), load the request into the second output logical data flow and send it;<br/>(g) if the third logical input data flow is received in (a), process the data it contains and check to see if this data provider has previously tried to load data into either of the NAP data stores and then carry out the checks in (c) above before updating internal records with the result;<br/>(h) when (f) is complete load the third logical output data flow with a "rejection" and if the checks in (g) were successful a "reminder to register" texts and send it to the destination obtained from the processing in (f).<br/>
