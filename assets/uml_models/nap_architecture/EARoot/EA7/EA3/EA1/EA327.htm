<b>“Data security and access restrictions for downloading”</b> follows the same logic as the previous one with the only difference being that data downloading is in the spotlight (instead of data uploading).<br/><br/>When accessing data as a consumer, are there security mechanisms in place to ensure the trust in the data or restricting the access to the NAP or the data?<br/><br/><b>Minimum acceptable value for interoperability: 3</b><br/><br/>Values:<br/>N/A: Metadata directory case (No data is provided via NAP online/web interface)<br/>0: No data is provided via NAP online/web interface in a data platform NAP type<br/>0: No security or authentication mechanisms are in place in a data platform NAP type<br/>0: Security and/or authentication mechanisms are in place to ensure the trust in the data or restrict the access for the NAP<br/>Value+1 (for each of the following options): Which security mechanisms are in place to ensure the trust in the data or restrict the access for your NAP? You can select more options:<br/>a. Transport security (https)<br/>b. Authentication by IP filter (access based on IP address of the consumer)<br/>c. Basic authentication according to RFC 7617 (https://tools.ietf.org/html/rfc7617)<br/>d. Digest Authentication according to RFC 7616 (https://tools.ietf.org/html/rfc7616)<br/>e. Authentication by url parameters<br/>f. Authentication by client certificate (private keys, certificates, incl. EU-wide common user authentication mechanism)<br/><br/>Minimum acceptable value for interoperability: 2<br/><br/>
